Texture Handling in the Flecs Engine

How the renderer turns a FlecsTexture component into a GPU sample: loading, normalization, material binding, and shader selection.

1. LoadPNG / JPG / DDS → WGPUTexture
2. SurveyAssign bucket + layer per material
3. BlitNormalize sources to RGBA8 slices
4. Mip-genCompute shader fills mips 1..N
5. SampleShader picks bucket, samples layer

1. Loading source textures

A texture is declared as a prefab-style entity with a FlecsTexture component that carries a file path: 

ECS_STRUCT(FlecsTexture, {
    const char *path;
});
include/modules/material.h

When FlecsTexture is set, the on_set hook flecsEngine_texture_onSet calls flecsEngine_texture_loadFile to produce a WGPUTexture and a matching WGPUTextureView, stored on a FlecsTextureImpl implementation component. 

typedef struct {
    WGPUTexture texture;
    WGPUTextureView view;
} FlecsTextureImpl;
src/types/components.h

Supported formats

SourceLoaderOutput formatMips
.dds (BC1/BC3/BC7, DX10 header) flecsEngine_texture_loadDds BCx{Unorm,Srgb} Preserved from file
.png, .jpg, .bmp stb_image → createFromPixels RGBA8Unorm Generated on CPU (box filter)
.png (fallback) tries sibling .dds file if PNG fails routes back into DDS loader

DDS parsing is minimal: only files that use the DX10 extended header are accepted, and only three DXGI formats map to a valid WGPUTextureFormat: 

BC1 UNORM / UNORM_SRGB   →  WGPUTextureFormat_BC1RGBAUnorm{,Srgb}
BC3 UNORM / UNORM_SRGB   →  WGPUTextureFormat_BC3RGBAUnorm{,Srgb}
BC7 UNORM / UNORM_SRGB   →  WGPUTextureFormat_BC7RGBAUnorm{,Srgb}
src/modules/renderer/dds.c

For compressed blocks the loader uses the DDS-reported mip_count and walks through the pixel data upload buffer mip by mip, sizing each copy as bytesPerRow = ((w + 3) / 4) * block_size with a block- aligned physical extent. BC1 uses 8-byte blocks, BC3/BC7 use 16-byte blocks.

Dimensions & shape

Every source texture is created as a 2D, single-layer WGPUTexture: 

.dimension = WGPUTextureDimension_2D,
.size = { width, height, .depthOrArrayLayers = 1 },
.sampleCount = 1,
.usage = TextureBinding | CopyDst | CopySrc

The view, however, is created with WGPUTextureViewDimension_2DArray — which is legal for a 1-layer texture and lets downstream code treat every texture uniformly when building the material texture arrays.

Source mips

  • DDS: the full mip chain present in the file is uploaded verbatim; the WebGPU texture is allocated with mipLevelCount = info.mip_count.
  • PNG/JPG: only mip 0 comes from the file. The loader computes mip_count = ceil(log2(max(w, h))) + 1, allocates a texture with that many levels, uploads mip 0 from stb_image, then synthesizes mips 1..N on the CPU with a 2×2 box filter: 
out[c] = (p00[c] + p10[c] + p01[c] + p11[c] + 2) / 4;
src/modules/renderer/render_textures.c
Why keep source mips at all? Because the blit pass that fills the material texture arrays (below) only samples mip 0 of the source. The source mip chain is not copied directly into the material arrays — it is only relevant for the very short-lived blit and could arguably be dropped for non-DDS loads. Keeping it makes the FlecsTextureImpl usable on its own for other systems (e.g. IBL, effects) that sample the source texture directly.

Fallbacks

Three 1×1 textures are created once per engine via flecsEngine_texture_create1x1: 

fallback_white_tex  = (255, 255, 255, 255)   // albedo, emissive, roughness
fallback_black_tex  = (  0,   0,   0, 255)
fallback_normal_tex = (128, 128, 255, 255)   // flat tangent-space normal

Each gets a 2DArray view. The normal fallback encodes the unit up-vector (0, 0, 1) after the shader's *2.0 - 1.0 unpack.

2. Materials and the GPU material buffer

A material is an ECS entity carrying a bundle of components: 

FlecsRgba        color            // base colour tint (rgba8)
FlecsPbrMaterial {metallic, roughness}
FlecsEmissive    {strength, color}
FlecsMaterialId  {value}          // dense index into the GPU buffer
FlecsPbrTextures {albedo, emissive, roughness, normal}  // optional
    // each channel is an ecs_entity_t pointing at a FlecsTexture

The renderer runs two queries against materials (src/modules/renderer/renderer.c):

  • materials.query — every entity with FlecsRgba + FlecsPbrMaterial + FlecsMaterialId (optional FlecsEmissive). Feeds the CPU FlecsGpuMaterial array.
  • materials.texture_query — entities that additionally have FlecsPbrTextures. Drives bucket assignment and the GPU blit pass.

The CPU side packs each material into a FlecsGpuMaterial, addressed by FlecsMaterialId.value: 

typedef struct {
    flecs_rgba_t color;           // base colour (4 bytes)
    float        metallic;
    float        roughness;
    float        emissive_strength;
    flecs_rgba_t emissive_color;
    uint32_t     texture_layer;   // slot inside its bucket array
    uint32_t     texture_bucket;  // 0 / 1 / 2  (512 / 1024 / 2048)
} FlecsGpuMaterial;
src/types/gpu_types.h

flecsEngine_material_uploadBuffer writes the packed array to a WGPUBufferUsage_Storage buffer. That buffer is plugged into the scene-globals bind group at @group(0) @binding(11), matching the WGSL declaration: 

@group(0) @binding(11)
var<storage, read> materials : array<GpuMaterial>;
src/modules/renderer/shaders/common/gpu_material_wgsl.h

All mesh vertices carry a per-instance material_id : u32 attribute. The vertex shader passes it through with @interpolate(flat) so every fragment in a draw call sees the same material index, which then indexes into the storage buffer.

Default bucket. The upload pass initializes every material with texture_bucket = 1. The texture-array build pass later overwrites this for materials that actually have FlecsPbrTextures; materials without textures stay on bucket 1 and sample the bucket-1 fallback arrays.

3. Normalization into bucketed texture arrays

PBR textured meshes never sample the source FlecsTextureImpl texture directly. Instead, all source textures that back a material are copied into one of three canonical 2DArray<rgba8unorm> textures per channel. This is what lets the shader bind a single bind group for the entire scene.

The bucket grid

bucket 0 bucket 1 bucket 2 (512x512) (1024x1024) (2048x2048) +---------+ +---------+ +---------+ albedo | 2DArray | | 2DArray | | 2DArray | @group(1) @binding(0..2) +---------+ +---------+ +---------+ emissive | 2DArray | | 2DArray | | 2DArray | @group(1) @binding(3..5) +---------+ +---------+ +---------+ roughness | 2DArray | | 2DArray | | 2DArray | @group(1) @binding(6..8) +---------+ +---------+ +---------+ normal | 2DArray | | 2DArray | | 2DArray | @group(1) @binding(9..11) +---------+ +---------+ +---------+ + 1 filtering sampler at @binding(12)

The bucket dimensions are fixed: 

static const uint32_t flecs_engine_bucket_dim[3] = { 512, 1024, 2048 };
#define FLECS_ENGINE_BUCKET_FORMAT WGPUTextureFormat_RGBA8Unorm
src/modules/renderer/render_materials.c

For each material that has textures, bucket selection picks the smallest bucket whose side length is ≥ the max side of the largest source channel. Anything larger than 2048 is clamped to bucket 2: 

max_dim = max(w_albedo, h_albedo, w_emissive, ..., w_normal, h_normal);
bucket  = smallest b such that bucket_dim[b] >= max_dim;

The survey pass (flecsEngine_textureArray_survey) walks every textured material, computes its bucket, and assigns it a dense slot within that bucket — then writes both into cpu_materials[mat_id] so the shader can find them.

Creating the bucket arrays

For every bucket that has at least one assigned slot, createBucket allocates 4 textures (one per channel): 

WGPUTextureDescriptor {
  .dimension       = 2D,
  .size            = { dim, dim, layer_count },
  .format          = RGBA8Unorm,
  .mipLevelCount   = log2(dim) + 1,       // full mip chain
  .usage           = TextureBinding | CopyDst
                   | RenderAttachment     // for blit pass
                   | StorageBinding       // for mip-gen compute
}

Every layer of every array is first pre-filled on mip 0 with the channel's neutral colour (white for albedo/emissive/ roughness, (128, 128, 255) for normal). Materials whose blit silently fails therefore still sample a sensible default instead of garbage memory.

Blit pass: source → bucket slot

The blit pass runs one fullscreen triangle per (material, channel), with a tiny inline WGSL pipeline: 

@group(0) @binding(0) var src_tex  : texture_2d<f32>;
@group(0) @binding(1) var src_samp : sampler;
@fragment fn fs_main(in : VertexOut) -> @location(0) vec4<f32> {
  return textureSample(src_tex, src_samp, in.uv);
}

The source is bound via a fresh 2D view into the source texture (so any input format — BC7, RGBA8, etc. — is decompressed by the sampler). The destination is a 2D view into one slice and mip 0 of the bucket array: 

// destination view: one layer, one mip of an array texture
WGPUTextureViewDescriptor {
  .dimension       = 2D,
  .baseArrayLayer  = slot,
  .arrayLayerCount = 1,
  .baseMipLevel    = 0,
  .mipLevelCount   = 1,
  .format          = RGBA8Unorm
}

A linear-filter sampler with ClampToEdge addressing is used so that a non-square source gets stretched to the square bucket without edge bleed. The result is that every source format, every source dimension, and every source shape is normalized to RGBA8Unorm at the bucket's canonical size.

Mip-gen compute pass

Mips 1..N of the bucket arrays are generated in a second pass by a compute shader. It reads mip N as a texture_2d_array<f32> and writes mip N+1 through a texture_storage_2d_array<rgba8unorm, write>: 

@group(0) @binding(0) var src : texture_2d_array<f32>;
@group(0) @binding(1) var dst : texture_storage_2d_array<rgba8unorm, write>;
@compute @workgroup_size(8, 8, 1)
fn cs_main(@builtin(global_invocation_id) id : vec3<u32>) {
  let layer   = i32(id.z);
  let src_xy  = vec2<i32>(id.xy * 2u);
  let s00 = textureLoad(src, src_xy + vec2<i32>(0,0), layer, 0);
  let s10 = textureLoad(src, src_xy + vec2<i32>(1,0), layer, 0);
  let s01 = textureLoad(src, src_xy + vec2<i32>(0,1), layer, 0);
  let s11 = textureLoad(src, src_xy + vec2<i32>(1,1), layer, 0);
  textureStore(dst, vec2<i32>(id.xy), layer, (s00+s10+s01+s11) * 0.25);
}

One dispatch is issued per (bucket, channel, destination mip), with workgroups covering the full layer count in the Z dimension. When the dispatch completes, each bucket array holds a full mip chain populated from the blitted mip 0. Source file mips are therefore thrown away — the final mips in the material array are always freshly box-filtered from the RGBA8 source.

When is the build triggered?

The per-frame system sequence in flecsEngine_renderer_render is: 

flecsEngine_material_uploadBuffer(world, impl);            // upload CPU materials
if (!impl->materials.texture_array_bind_group) {           // only if missing
    flecsEngine_material_buildTextureArrays(world, impl);  // survey + create + blit + mipgen + bindgroup
}

A material upload that detects a changed next_id calls flecsEngine_textureArray_release, which clears the bind group pointer and forces the arrays to be rebuilt on the next frame. Steady-state frames therefore skip the build entirely.

4. Binding

Bind group layout (group 1)

The layout is built once and cached in pbr_texture_bind_layout: 

bindings 0..11:  12 × texture_2d_array<f32> (float-sampleable, visible to fragment)
binding     12:  1 × filtering sampler        (visible to fragment)
src/modules/renderer/bind_groups/textures.c

The binding index encodes (channel, bucket) in row-major order: binding = channel * 3 + bucket.

Assembling the bind group

createBindGroup creates a 2DArray view that spans all layers and all mips for every populated bucket. If a particular bucket was never populated, its slot is backfilled with the 1×1 fallback view (white for channels 0–2, normal-blue for channel 3), so the layout is always fully satisfied: 

for ch in 0..4:
    fb = fallback_normal if ch == 3 else fallback_white
    for b in 0..3:
        views[ch*3 + b] = buckets[b].texture_array_views[ch] ?? fb

The sampler at binding 12 is shared with the rest of the PBR pipeline: trilinear linear filtering (min/mag/mip all Linear), repeat addressing, max anisotropy 16, LOD clamp [0, 32].

Per-draw binding

A textured mesh batch binds the full scene-wide texture group once at the top of its render callback and never rebinds it: 

if (!engine->materials.texture_array_bind_group) return;
wgpuRenderPassEncoderSetBindGroup(
    pass, 1, engine->materials.texture_array_bind_group, 0, NULL);

// ...iterate groups (one per mesh), draw each instance buffer.
src/modules/renderer/batches/mesh.c

The transparent mesh batch does the same thing on demand when a sorted instance is marked is_textured — it switches to the textured pipeline and sets the same single bind group. One bind group serves every textured draw in the scene.

5. Shader-side selection

The fragment shader for pbr_textured has three problems to solve per pixel:

  1. Find this fragment's material in the global material buffer.
  2. Pick the correct size bucket — which binding to sample.
  3. Pick the correct array slice inside that bucket.
@fragment fn fs_main(input : VertexOutput) -> @location(0) vec4<f32> {
    let material = materials[input.material_id];      // storage lookup
    let layer    = material.texture_layer;             // slice inside bucket
    let bucket   = material.texture_bucket;            // 0 / 1 / 2
    let dx = dpdx(input.uv);                           // lifted derivatives
    let dy = dpdy(input.uv);
    let base_color = sample_albedo(input.uv, layer, bucket, dx, dy);
    // ... roughness, emissive, normal same way
}
src/modules/renderer/shaders/pbr_textured.c

The per-channel sample helpers

Each of the four channels has a helper that picks one of three array textures with a switch on bucket: 

fn sample_albedo(uv : vec2<f32>, layer : u32, bucket : u32,
                 dx : vec2<f32>, dy : vec2<f32>) -> vec4<f32> {
  switch (bucket) {
    case 0u: { return textureSampleGrad(albedo_tex_512,  tex_sampler, uv, layer, dx, dy); }
    case 1u: { return textureSampleGrad(albedo_tex_1024, tex_sampler, uv, layer, dx, dy); }
    default: { return textureSampleGrad(albedo_tex_2048, tex_sampler, uv, layer, dx, dy); }
  }
}
Why textureSampleGrad and not textureSample? Because the bucket value comes from a storage buffer lookup and is non-uniform across a 2×2 quad when neighbouring fragments belong to different materials. Calling textureSample from inside a non-uniform control-flow branch is a WGSL error — the implicit derivatives would be undefined. By computing dpdx(uv) and dpdy(uv) once at the top of fs_main, before any branching, and passing them in explicitly, the derivatives are always valid and the sampler can still choose the correct mip level. This is what lets neighbouring fragments using different buckets render correctly on the same quad.

Mip level selection

Mip selection is entirely the sampler's job. Given the uv gradients (dx, dy) the hardware computes a LOD and interpolates between the two nearest mip levels (trilinear), combining across up to 16 anisotropic taps. The LOD is clamped to [0, 32], which is effectively "no clamp" given a 2048-wide array (~11 mips).

Because every bucket array has a full mip chain generated by the compute pass, the sampler has a usable level for any projected footprint, all the way down to a 1×1 final mip.

Flat interpolation of material_id

The vertex input carries a per-instance @location(8) material_id : u32. It is output from the vertex stage with @interpolate(flat), which guarantees every fragment in a triangle gets the provoking vertex's value verbatim. This is what lets the per-fragment materials[input.material_id] lookup produce a single consistent bucket/layer pair for the triangle.

Putting it together

The end-to-end path for a single sampled texel:

FlecsTexture.path [ECS component] | | on_set hook → flecsEngine_texture_loadFile v WGPUTexture (source, 2D, 1 layer, RGBA8 or BCx, full mips) [FlecsTextureImpl] | | --- material upload phase --- | | CPU materials array built from FlecsPbrMaterial, FlecsRgba, ... | cpu_materials[mat_id] = { color, metallic, ..., bucket=1 } (default) | | --- texture array build phase (only when materials change) --- | | survey: for each material with FlecsPbrTextures, | choose bucket by max side, assign dense slot | → cpu_materials[mat_id].{texture_bucket, texture_layer} | | create: buckets[b].texture_arrays[ch] = 2DArray<rgba8unorm>, full mips | fill: mip 0 prefilled with per-channel fallback colour | | blit: per (material, channel) → fullscreen-triangle render | source view (2D, mip 0) ⇒ dest view (slice, mip 0) | | mipgen: compute shader, one dispatch per (bucket, channel, mip) | reads 2DArray mip N, writes 2DArray mip N+1 (rgba8unorm) | | bindgr: @group(1) = 12 × 2DArray views + 1 sampler v Render pass | | bind @group(0) scene globals (materials storage @binding(11)) | bind @group(1) PBR texture arrays (once per frame per pass) | | vertex shader: forward material_id with @interpolate(flat) | | fragment shader: | mat = materials[material_id] (storage buffer lookup) | layer = mat.texture_layer | bucket = mat.texture_bucket | dx, dy = dpdx(uv), dpdy(uv) (lifted from branch) | switch(bucket) → textureSampleGrad(albedo_tex_{512,1024,2048}, ...) v Final RGBA sample
